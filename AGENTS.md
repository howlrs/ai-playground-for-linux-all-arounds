# エージェント連携定義

## 概要

本環境では3つのAIエージェントが連携して開発を行う。
それぞれの役割は明確に分離されており、相互に補完し合う。

## エージェント定義

### Claude Code - メインエージェント

```
役割:     設計・実装・統括
起動:     claude
権限:     コンテナ内フルアクセス + 外部CLI（ポリシー準拠）
判断権限:  アーキテクチャ決定、最終的な実装判断
```

**責務:**
- ユーザーとの対話による要件定義
- 設計文書・実装計画の作成
- コードの実装とテスト
- Gemini CLIへのQA依頼と結果の判断
- Codexへのタスク委任
- 外部サービスCLI操作（ポリシー準拠）

**判断基準:**
- Gemini CLIのフィードバックは参考にするが、技術的根拠を自ら検証する
- Codexの出力は必ずレビューしてから採用する
- 不明点はユーザーに確認する（推測で進めない）

---

### Gemini CLI - QAエージェント

```
役割:     品質保証・レビュー・別視点の提供
起動:     gemini
権限:     読み取り専用（コード閲覧、分析）
判断権限:  なし（提案のみ）
```

**QA依頼の仕組み:**

Gemini CLIは `-p`（一時prompt）モードで呼び出す。プロジェクトルートの `GEMINI.md` が
毎回自動読み込みされるため、QA役割・レビュー観点・回答フォーマットは前提として
設定済み。依頼時は**対象と種別のみ**を簡潔に渡す。

#### 依頼テンプレート
```bash
# コードレビュー
gemini -p "コードレビュー依頼:
$(git diff main...HEAD)"

# 設計レビュー
gemini -p "設計レビュー依頼:
$(cat docs/superpowers/specs/YYYY-MM-DD-*.md)"

# テストレビュー
gemini -p "テストレビュー依頼:
実装: $(cat src/main.ts)
テスト: $(cat tests/main.test.ts)"

# セキュリティレビュー
gemini -p "セキュリティレビュー依頼:
$(cat src/auth.ts)"

# 特定観点を指定する場合
gemini -p "コードレビュー依頼（パフォーマンス重点）:
$(git diff HEAD~3...HEAD)"
```

#### 回答フォーマット（GEMINI.mdにて定義済み）
Gemini CLIは以下の構造で回答する：
- **Critical（対応必須）**: 問題箇所 + 影響 + 修正案
- **Important（対応推奨）**: 問題箇所 + 影響 + 修正案
- **Suggestion（提案）**: 改善案
- **総評**: 全体的な品質評価

#### レビュー結果の扱い方
- Critical: 必ず対応する。ただし技術的根拠を自ら検証すること
- Important: 根拠を検証した上で対応を判断する
- Suggestion: 参考にする。対応は任意
- 根拠が不明な指摘: 鵜呑みにせず、自分で調査して判断する

### OpenAI Codex - 補助エージェント

```
役割:     単純タスクの並列処理
起動:     codex
権限:     コンテナ内ファイル操作
判断権限:  なし（指示された作業のみ）
```

**委任してよいタスク:**
- コードフォーマット・リント修正
- 型定義ファイルの生成
- ボイラープレートコードの作成
- ドキュメントの定型部分の生成
- 単純なリファクタリング（変数名変更等）

**委任してはいけないタスク:**
- アーキテクチャに影響する設計判断
- セキュリティに関わる実装
- 外部サービスCLI操作
- テスト戦略の決定

## 連携フロー

### 通常の開発フロー

```
ユーザー
  │
  ▼
Claude Code (メイン)
  │
  ├─── 設計フェーズ ──→ Gemini CLI (設計レビュー)
  │                        │
  │    ◄── フィードバック ──┘
  │
  ├─── 実装フェーズ ──→ Codex (単純タスク委任)
  │                        │
  │    ◄── 成果物 ────────┘
  │
  ├─── QAフェーズ ───→ Gemini CLI (コードレビュー)
  │                        │
  │    ◄── レビュー結果 ──┘
  │
  ├─── 修正・検証
  │
  ▼
完了報告 → ユーザー
```

### バグ修正フロー

```
ユーザー: "バグ報告"
  │
  ▼
Claude Code
  ├─ 1. systematic-debugging で原因調査
  ├─ 2. 仮説を立てる
  ├─ 3. Gemini CLI に仮説の妥当性を確認
  │     gemini -p "この仮説は妥当ですか？ ..."
  ├─ 4. テストを書く（RED）
  ├─ 5. 修正を実装（GREEN）
  ├─ 6. Gemini CLI に修正をレビュー依頼
  ├─ 7. verification-before-completion で検証
  └─ 8. 完了報告
```

### 大規模機能開発フロー

```
ユーザー: "新機能要件"
  │
  ▼
Claude Code
  ├─ 1. brainstorming で設計
  │     → Gemini CLI に設計レビュー依頼
  ├─ 2. writing-plans で計画作成
  ├─ 3. using-git-worktrees でブランチ作成
  ├─ 4. タスクごとに実装
  │     ├─ 単純タスク → Codex に委任
  │     └─ 複雑タスク → Claude Code が実装
  ├─ 5. 実装完了 → Gemini CLI にフルレビュー
  ├─ 6. フィードバック対応
  ├─ 7. verification-before-completion
  └─ 8. finishing-a-development-branch
```

## QA品質基準

Gemini CLIへのレビュー依頼時、以下の観点を含めること：

| 観点 | 確認内容 |
|------|---------|
| 正確性 | ロジックにバグはないか |
| セキュリティ | OWASP Top 10に該当する脆弱性はないか |
| パフォーマンス | 不要なループ、N+1問題、メモリリークはないか |
| 可読性 | 変数名・関数名は適切か、複雑度は許容範囲か |
| テスト | エッジケース、異常系は網羅されているか |
| 設計 | SOLID原則に反していないか、責務は適切に分離されているか |
